## 1. 배경 (Context)

온라인 쇼핑몰 대표 "토토"는 주문량 급증으로 **주문·회계·운송**이 수기로 연계되어 누락/지연이 발생한다.
EAI 전문기업 인스피언에 다음 자동화를 의뢰했다.

1. 결제 완료 주문을 **주문 DB에 적재**함과 **동시에** 회계 담당자에게 **영수증 파일(FTP)** 전송
2. 적재된 주문을 **5분 주기 배치**로 운송사 DB에 전달 → 배송 접수 자동화

지원자는 인스피언 개발자로서 **EAI 특성(Sender→Mapper→Receiver)** 을 고려해 API 기반 연계 시스템을 설계·구현한다.

## 3. 범위 (Scope)

### 3.1 In Scope
- **S1**: 실시간 주문 처리 (REST API, 동기)
- **S2**: 운송사 전송 배치 (스케줄러 5분 주기)
- **S3**: 공통 — 운영 로그(로컬 파일) + JDBC/FTP 예외 처리
- **S4**: 아키텍처 변화 다이어그램 (제출 전/후)
- **Bootstrap**: 제공 API 호출 + AES 복호화로 접속정보 확보 (S1·S2의 전제조건)

### 3.2 Out of Scope (YAGNI)
- 인증/인가, 사용자 관리, 화면(UI)
- 실제 운송사/회계 시스템 연동 (제공된 FTP/DB가 종착점)
- 메시지 큐·분산 트랜잭션 코디네이터 등 과제 범위를 넘는 인프라
- 다중 인스턴스 수평 확장 (단일 인스턴스 가정, 단 채번 동시성은 단일 인스턴스 내에서 방어)

## 4. 용어 (Glossary)

| 용어 | 정의 |
|------|------|
| Sender | 인스피언 서비스로 들어오는 요청의 연결부 (S1의 REST 수신) |
| Receiver | 인스피언 서비스에서 나가는 데이터의 연결부 (JDBC/FTP) |
| Mapper | 요청 데이터를 수신 측 스키마에 맞게 변환하는 부분 |
| APPLICANT_KEY | 제공 API로 받는 지원자 고정 키. 모든 적재행의 PK 일부로 고정 삽입 |
| STATUS 'N'/'Y' | 주문의 운송 전송 여부. 'N'=미전송(신규), 'Y'=전송완료. **S1↔S2의 연결고리** |

## 5. 연계 흐름 요구사항 (Flow)

```
[S1 실시간]
주문자 ──REST(XML)──▶ Sender ──▶ 검증 ──▶ Mapper(1:N→flat)
                                              ├─▶ Receiver(JDBC) ──▶ ORDER_TB
                                              └─▶ Receiver(FTP)  ──▶ 영수증 .txt
                          ◀──JSON 처리결과──── [JDBC AND FTP 모두 성공 시에만]

[S2 배치 5분]
Scheduler ──▶ ORDER_TB(STATUS='N' 조회) ──▶ Mapper ──▶ SHIPMENT_TB 적재
                                                         └─▶ 성공건 ORDER_TB.STATUS='Y'
```

| ID | 요구사항 | 축 |
|----|----------|-----|
| FR-FLOW-01 | 모든 연계는 Sender→Mapper→Receiver 구조로 분리·구현한다. | A2 |
| FR-FLOW-02 | S1의 JDBC와 FTP는 **둘 다 성공해야** 주문자에게 성공을 반환한다(동기화). | A2,A4 |
| FR-FLOW-03 | S2는 STATUS='N'을 입력으로, STATUS='Y' 갱신을 출력으로 하여 S1과 결합한다. | A2 |

## 6. 시나리오 1 — 실시간 주문 처리 (REST, 동기)

### 6.1 입력 (Request)
- 프로토콜: REST (POST), 본문: 주문 XML (HEADER:ITEM = **1:N**)
- HEADER: `USER_ID, NAME, ADDRESS, STATUS`
- ITEM: `USER_ID, ITEM_ID, ITEM_NAME, PRICE`
- HEADER와 ITEM은 `USER_ID`로 조인하여 flat 구조로 매핑

### 6.2 기능 요구사항

> **출처** 컬럼: `[스펙]` = PPTX에 명시된 요구사항 / `[설계]` = 스펙엔 없으나 운영·정합성을 위해 우리가 정의한 보조 요구사항(`-a` 접미사).
> `-a` 행은 바로 위 모(母) 요구사항에 딸린 파생 결정이다.

| ID | 요구사항 | 출처 | 축 |
|----|----------|------|-----|
| FR-S1-01 | 주문 XML을 수신하고 유효성을 검증한다(필수필드 누락/형식). | [스펙] | A3,A4 |
| FR-S1-01-a | 검증 규칙(필수 필드·형식)을 §6.3 표대로 적용한다. | [설계] | A3,A4 |
| FR-S1-02 | HEADER:ITEM(1:N)을 USER_ID로 조인해 flat 레코드로 변환한다. | [스펙] | A3 |
| FR-S1-03 | 변환 레코드를 ORDER_TB에 JDBC로 적재한다. | [스펙] | A2,A3 |
| FR-S1-03-a | 적재 시 STATUS는 입력값과 무관하게 `'N'`으로 고정한다(이상 입력 방어). | [설계] | A4 |
| FR-S1-04 | ORDER_ID를 채번한다. 형식: **대문자 1 + 숫자 3** (예: `A113`). 자동채번 아님. | [스펙] | A3 |
| FR-S1-04-a | 채번 증가·충돌 규칙은 §8.4(NFR-ID)를 따른다(동시성·재시도). | [설계] | A4 |

| FR-S1-07 | 파일명: `INSPIEN_{참여자명(한글)}_{yyyyMMddHHmmss}.txt` | [스펙] | A3 |
| FR-S1-08 | 파일 내용 1행 포맷(구분자 `^`, 행 끝 `\n`): `ORDER_ID^USER_ID^ITEM_ID^APPLICANT_KEY^NAME^ADDRESS^ITEM_NAME^PRICE`. 필드·값·순서 유지. | [스펙] | A3 |
| FR-S1-09 | FR-S1-03(JDBC)과 FR-S1-06(FTP)이 모두 완료된 뒤 처리 결과(JSON)를 반환한다(SYNC). | [스펙] | A2,A4 |
| FR-S1-09-a | 응답 JSON 스키마(성공/실패)를 §6.4대로 정의한다. | [설계] | A4 |

### 6.3 입력 검증 규칙 (FR-S1-01-a 상세)

> 원본 스펙은 "유효성 검증"만 명시. 아래 규칙은 우리가 정의한 [설계] 항목이다.

| NAME | HEADER | ✅ | 공백 불가 | 거부 |
| ADDRESS | HEADER | ✅ | 공백 불가 | 거부 |
| STATUS | HEADER | ⛔ | 입력값 무시(§FR-S1-03-a로 'N' 고정) | — |
| ITEM_ID | ITEM | ✅ | 공백 불가 | 거부 |
| ITEM_NAME | ITEM | ✅ | 공백 불가 | 거부 |
| PRICE | ITEM | ✅ | 숫자 문자열(정수) | 거부 |
| (구조) | — | ✅ | 모든 ITEM의 USER_ID가 HEADER에 존재 | 거부 |

## 9. Bootstrap — 제공 정보 수신 (전제조건)

> S1·S2 모든 작업의 출발점. DB/FTP 접속정보를 제공 API에서 받아 복호화해야 한다.

| ID | 요구사항 | 출처 | 축 |
|----|----------|------|-----|
| FR-BOOT-01 | 제공 API에 개인정보(이름/전화/이메일)를 POST하여 접속정보를 수신한다. | [스펙] | A2 |
| FR-BOOT-02 | 응답의 ORDER_TB_CONN/SHIPMENT_TB_CONN/FTP_CONN을 AES-128로 복호화한다. | [스펙] | A3 |
| FR-BOOT-03 | SAMPLE_DATA(base64·EUC-KR XML)를 디코딩해 S1 입력 샘플로 활용한다. | [스펙] | A3 |
| FR-BOOT-03-a | 복호화한 접속정보는 메모리에서만 사용하고 로그·커밋·예외 메시지에 노출하지 않는다. | [설계] | A4 |

### 9.1 외부 인터페이스
- **호출**: `POST https://inspien.it-cpi002-rt.cfapps.ap10.hana.ondemand.com/http/RecruitingTest`
- **인증**: Basic Auth (자격증명은 `application-secret`로 분리, 본 문서 미기재)
- **Request**: `{"NAME","PHONE_NUMBER":"010-XXXX-XXXX","E_MAIL"}` — 복붙 금지(글자깨짐), 직접 입력
- **Response**: `{APPLICANT_KEY, ORDER_TB_CONN, SHIPMENT_TB_CONN, FTP_CONN, SAMPLE_DATA}`

### 9.2 복호화 규칙 (A3 정합성 핵심)
- 알고리즘: **AES-128 / ECB / PKCS5Padding**
- Key 생성: `휴대폰번호 문자열 → UTF-8 bytes → SHA-1 해싱 → 앞 16바이트(128bit)`
- 암호문은 Base64 인코딩 → **Base64 디코딩 후 복호화**
- SAMPLE_DATA: Base64 디코딩 시 **EUC-KR** XML (인코딩 주의)

## 10. 제약 / 가정 (Constraints & Assumptions)

> `C-`(Constraint)는 지켜야 할 제약, `A-`(Assumption)는 우리가 둔 가정. 출처로 [스펙]/[설계] 구분.

| ID | 항목 | 출처 |
|----|------|------|
| C-01 | 개인정보(이름·전화·이메일)와 참여자명(한글)은 비밀값으로 코드/문서에 하드코딩하지 않고 외부 설정으로 분리한다. | [설계] |
| C-02 | DBMS 종류는 복호화로 접속정보 확보 후 확정되며, 그때 JDBC 드라이버를 추가한다. | [스펙] |
| C-03 | 화면·Web App Platform은 불필요. 필요한 라이브러리만 사용한다. | [스펙] |
| C-04 | 작업 단위 커밋 이력을 남긴다(컨벤셔널 커밋). | [스펙] |
| A-01 | 단일 인스턴스 실행을 가정한다(수평 확장 미고려, 단 인스턴스 내 동시성은 방어). | [설계] |

## 11. 수용 기준 (Acceptance Criteria)

데모(면접 시연) 기준 — 참여자명 + 당일 날짜로 조회 가능해야 한다.

| ID | 기준 |
|----|------|
| AC-01 | 제공 API 호출 → 복호화 → DB/FTP 접속정보 확보가 재현된다. |
| AC-02 | S1: 샘플 XML POST 시 ORDER_TB에 행이 적재되고(STATUS='N'), 규격 파일이 FTP에 업로드된다. |
| AC-03 | AC-02에서 DB와 FTP 결과가 **일관**된다(부분성공 없음). |
| AC-04 | S2: 5분 배치 실행 후 STATUS='N' 건이 SHIPMENT_TB에 적재되고 원본 STATUS가 'Y'로 바뀐다. |
| AC-05 | 처리 단위 로그가 로컬 파일에 성공/실패로 남는다. |
| AC-06 | JDBC/FTP 강제 실패 시(연결 차단 등) 예외가 처리되고 로그에 실패로 기록된다. |
| AC-07 | DBMS 접속 후 참여자명·당일날짜 조회, FTP 접속 후 업로드 파일 확인이 가능하다. |
